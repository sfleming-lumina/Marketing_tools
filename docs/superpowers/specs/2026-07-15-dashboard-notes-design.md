# Dashboard Element Notes — Design

**Date:** 2026-07-15
**Status:** Approved
**Author:** Sean Fleming (with Claude)

## Problem

The Marketing Decision Workbench (`outputs/marketing_decision_tool.html`) is deployed on Cloud Run for the marketing team to explore. There's currently no way for the team to leave feedback on specific parts of the dashboard, which makes it hard to collect input for refining the tool toward their actual use case. Feedback today would have to happen out-of-band (email, Slack, verbally), disconnected from the specific chart/metric/card it's about.

## Goal

Let marketing team members select an element on the dashboard (a metric tile, chart panel, campaign card, decision-queue item, or BQ object row) and attach a note to it. Notes are centralized (not per-browser) so the whole team, and whoever maintains the tool, can see them in one place.

## Non-goals

- No authentication/login. Attribution is a self-reported display name, not a verified identity — acceptable for a small internal team tool.
- No editing or deleting existing notes. Corrections happen as follow-up notes (append-only history).
- No real-time sync (websockets/polling). Notes refresh on page load and after submitting.
- No resolved/status workflow. Triage happens by reading the Feedback view or querying BigQuery directly.

## Architecture

Two new pieces, isolated from whatever currently deploys the static site so that deployment isn't disturbed:

1. **`notes-api` service** — a small Python/FastAPI app, deployed as its own Cloud Run service (separate from the existing static-file Cloud Run service). Endpoints:
   - `GET /notes?view=<optional>` — list notes, optionally filtered by dashboard view.
   - `POST /notes` — insert a new note.
   CORS is opened to the dashboard's origin. The service account it runs as needs `BigQuery Data Editor` on the notes table.

2. **BigQuery table** — `lumina-lakehouse.marketing_tool_ops.dashboard_notes` (new dataset). Append-only: every note is an `INSERT`, never an `UPDATE`/`DELETE`. This sidesteps BigQuery's streaming-buffer restrictions on recently-inserted rows and keeps a full history of feedback.

   Schema:
   | column | type | notes |
   |---|---|---|
   | note_id | STRING | UUID, generated by the API |
   | created_at | TIMESTAMP | server-assigned |
   | author_name | STRING | self-reported, from the browser's stored display name |
   | view | STRING | `overview` \| `cohorts` \| `campaigns` \| `scenario` \| `objects` |
   | element_key | STRING | stable semantic id, e.g. `campaign:Paid Search:Google Nonbrand Search` |
   | element_label | STRING | human-readable label shown in the UI, e.g. "Google Nonbrand Search campaign card" |
   | note_text | STRING | the note itself |
   | context | STRING (JSON) | active filters at time of writing: market, source, window, and view-specific state (objective/grain/selected campaign, scenario slider values) |

3. **Frontend** — only `outputs/marketing_decision_tool.html` changes. A `NOTES_API_BASE` constant points at the new service. Notes are fetched on load and after a successful submit.

## Element selection & stable keys

Two levels of granularity:
- **Panels** — each chart/table section with its own `<h3>` heading.
- **Repeated cards** — each metric tile, campaign card, decision-queue item, and BQ object row, individually.

Every annotatable element gets a small note-affordance icon (visually consistent with the existing `.help-chip` "?" icon, placed at the opposite corner so the two never collide) showing a badge with the note count when > 0. Clicking it opens a single reusable slide-in drawer (right side of the screen) showing existing notes for that element plus a compose box for a new one.

Element keys are derived from the underlying data the render functions already have, not array position/index, so notes stay attached to the right thing when filters reorder or hide cards:
- Metrics/charts (fixed): `metric:<slug>`, `chart:<id>`, `panel:<slug>`
- Campaign cards: `campaign:<source>:<campaign name>`
- Decision-queue items: `decision:<slug of title>`
- Object rows: `object:<table name>`

## Attribution

No login exists. The first time someone opens the notes drawer to compose a note, they're asked for a display name, stored in `localStorage` and reused for all future notes from that browser. A small "not you? change name" control lives in the drawer footer.

Each note also captures the dashboard's active filter state at time of writing (market/source/window, plus campaign-view objective/grain/selection and scenario slider values), since "this number looks off" is meaningless without knowing what was on screen.

## New "Feedback" view

A sixth nav item, **Feedback**, lists every note across the dashboard in one place, filterable by `view`. This is the primary triage surface for the tool maintainer, though the BigQuery table can also be queried directly.

## Error handling

If `notes-api` is unreachable:
- Existing notes simply fail to load (empty state, not a page-breaking error).
- Composing a note still works in the UI; the submission is queued in `localStorage` and shows "not saved yet — will retry," retried on next page load. The rest of the (otherwise static/offline-tolerant) dashboard is unaffected.

## Testing

This is a static-HTML-plus-tiny-API project, so testing is mostly manual/integration rather than unit-test-heavy:
- API: basic tests for `POST /notes` (valid insert, required-field validation) and `GET /notes` (filtering).
- End-to-end: drive the actual page — add a note to a metric, a campaign card, and an object row; reload; confirm all three persist and appear correctly attributed and filtered in the Feedback view.
- Verify the offline/unreachable-API fallback path (queue + retry) manually by pointing `NOTES_API_BASE` at a dead URL.

## Deployment notes (for the implementation plan)

- New Cloud Run service (`notes-api`) needs to be created and deployed; region should match the existing dashboard service for latency.
- New BigQuery dataset (`marketing_tool_ops`) and table need to be created; the plan should include the exact DDL.
- The `notes-api` service account needs `BigQuery Data Editor` on the new table (or dataset).
- After editing `outputs/marketing_decision_tool.html`, the existing static-site deployment pipeline (not touched by this project) will need to be re-run to publish the change — that step is on the user since the pipeline's source isn't part of this repo.
